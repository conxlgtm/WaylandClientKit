# Identity model

WaylandClientKit exposes identity as platform data. Public APIs
should expose stable concrete identities that let callers correlate snapshots of
the same logical entity without confusing identity with current state.

The model is intentionally not one generic ID type. Different domains carry
different lifetime, scope, and construction rules.

## Taxonomy

### Raw protocol identity

Raw protocol identity comes from Wayland protocol objects, registry names, or
protocol serials. These values correspond to compositor facts or raw wrapper
lifecycle and may be useful for logging, correlation, or protocol round trips.

Examples include raw object identifiers, registry names, `SeatID`, and
`InputSerial`.

### Application identity

Application identity comes from the app and is passed through WaylandClientKit to
the compositor. Apps need access to the stored value so they can create and reuse
the same identity where the protocol calls for it.

An example is `ActivationAppID`.

### WaylandClientKit client identity

WaylandClientKit client identity is generated by WaylandClientKit for managed state.
These IDs are stable for the lifetime of the managed entity, but they are not
protocol facts and callers should not persist or construct them.

Examples include `WindowID`, `PopupID`, `DataOfferID`, `DataSourceID`,
`PointerConstraintID`, `RelativePointerSubscriptionID`, and presentation
feedback IDs.

### Public projection identity

Public projection identity exposes a stable public domain while hiding shared
internal storage domains. Projection identities are useful when multiple public
features share one internal ID space but should not leak that implementation
detail.

Examples include `PopupSurfaceIdentity`, `ClipboardOfferIdentity`,
`PrimarySelectionOfferIdentity`, `DragOfferIdentity`,
`ClipboardSourceIdentity`, `PrimarySelectionSourceIdentity`, and
`DragSourceIdentity`.

### Display-owned handle identity

Display-owned handle identity pairs an entity ID with the owning display
identity. Handles with the same entity ID but different displays are different
logical handles and must not compare as equal.

Examples include `Window`, `PopupSurface`, data-transfer offers and sources,
`TextInputSession`, `RelativePointerSubscription`, and `PointerConstraint`.

### Seat-scoped identity

Seat-scoped identity combines a `SeatID` with resources whose validity depends
on that seat. Seat removal must make those dependent resources stale in a
deterministic way.

Examples include text-input sessions, relative pointer subscriptions, pointer
constraints, and selection resources.

### Opaque protocol token

Opaque protocol tokens are payloads provided by protocols and passed back to the
compositor or another process. They are stable as protocol payloads, not as
WaylandClientKit-generated resource IDs.

Examples include `ActivationToken`.

## Description policy

Public ID descriptions should be stable, lowercase, hyphenated, and
domain-prefixed, such as `window-1`, `seat-1`, `serial-1`, and
`clipboard-offer-1`.

Descriptions are for diagnostics and logs. They are not parseable API contracts.

## Raw value visibility

Expose `rawValue` publicly only when the value is a protocol-level fact the app
may need to preserve, compare with external data, or log.

Keep `rawValue` package-scoped when the value is WaylandClientKit-generated and
should not be persisted or constructed by users.

Current examples:

- `SeatID.rawValue` is public because the value is a protocol-visible seat
  identity.
- `InputSerial.rawValue` is public because the value is a protocol serial.
- `WindowID.rawValue` is package-scoped because it is generated by
  WaylandClientKit.
- `PopupID`, `DataOfferID`, `DataSourceID`, pointer-capture IDs, diagnostics,
  foreign-toplevel IDs, output-management head/mode IDs, presentation IDs, and
  projection identity raw values are package-scoped because they are
  implementation identities.
- `ActivationToken.value` is public because the token string is the protocol
  payload.

## Identifiable policy

Use `Identifiable` when a public type already has an entity identity that
correlates the same logical resource across state snapshots.

The `id` must be the entity identity, not a transient runtime detail, request
counter, callback object, or complete state snapshot.

## Package support

Package-internal identity helpers define the common shape for IDs and ID
generation without turning those helpers into public API:

- `WaylandEntityID`
- `UInt64WaylandEntityID`
- `UInt32WaylandEntityID`
- `PrefixedIdentityDescription`
- `IDGenerator`

These helpers are implementation infrastructure. Public users should continue to
use stable concrete identity types.

## Checked-in identity declarations

`identities/generation-policy.json` is the source of truth for repetitive
one-field identity wrappers. It records each concrete type, its access and raw
value rules, conformances, diagnostic description, and any narrow projection or
integer-literal behavior. The generator emits separate checked-in files for the
public and internal declarations in `WaylandClient`, `WaylandRaw`, and
`WaylandGraphicsPreview`. It does not replace these domain-specific types with a
generic ID.

Generate and verify the files with:

```bash
swift run wck identity generate
swift run wck identity verify-generated
```

Generated public identities also carry their audit category in this policy.
Constructor and stored-value visibility come from the same declaration fields.
`identities/manual-identity-categories.json` contains only the four public
identities whose declarations remain handwritten. Generation combines both
inputs into `docs/identity-categories.json`, so the visibility audit does not
duplicate generated declaration facts in a second handwritten manifest.

The generated [identity visibility audit](identity-visibility.md) records every
public identity category together with its constructor and stored-value access.
After an intentional identity contract change, run
`swift run wck identity verify --update` to refresh that report.
`swift run wck ci cheap` verifies the generated declarations, combined category
manifest, and visibility report against the Swift declarations.
